The right mental model

If you've tested web applications before, you already understand components. Think of LWC like a React or Angular component — a self-contained unit with:

  • Template (HTML) — what renders in the browser
  • Controller (JS) — state and behaviour
  • Style (CSS) — scoped styles, never leak out
🌍 WFP Context

The Donor Portal is built on Salesforce Experience Cloud. Donors log in, view their giving history, download tax receipts, and manage their recurring gifts. Every screen is an LWC — or a composition of several LWCs.

Module 1

LWC Fundamentals

Every LWC lives in a folder. That folder always contains three files with the same name. Understanding this structure is 80% of reading any component.

The Component Bundle

donorSummary/
  ├── donorSummary.html   ← template (what the user sees)
  ├── donorSummary.js    ← controller (logic & state)
  ├── donorSummary.css   ← scoped styles
  └── donorSummary.js-meta.xml ← metadata (where it can deploy)

The HTML Template

LWC templates always wrap in <template>. Inside, it's mostly regular HTML — but with a few Salesforce directives for conditionals and loops.

<template>
  <!-- Conditional rendering -->
  <template if:true={isLoggedIn}>
    <p>Welcome back, {donorName}</p>
  </template>

  <!-- Loop -->
  <template for:each={gifts} for:item="gift">
    <p key={gift.Id}>{gift.Amount__c}</p>
  </template>

  <!-- Event handler -->
  <button onclick={handleDownload}>Download Receipt</button>
</template>
donorSummary.html
Key pattern

Curly braces {variableName} bind to JS properties. If the JS property changes, the HTML updates automatically — no DOM manipulation needed.

The JavaScript Controller

Every JS file exports a class that extends LightningElement. Properties you want bound to the template just need to be declared — or decorated.

import { LightningElement, track, api, wire } from 'lwc';

export default class DonorSummary extends LightningElement {

  // @api — public prop, set by parent component
  @api donorId;

  // @track — reactive private state (object/array)
  @track gifts = [];

  // plain property — also reactive for primitives
  donorName = '';
  isLoggedIn = false;

  // Lifecycle hook — runs after component renders
  connectedCallback() {
    console.log('Component mounted');
  }

  // Event handler
  handleDownload() {
    // fire an event, call Apex, etc.
  }
}
donorSummary.js

Decorators at a glance

DecoratorPurposeDirection
@apiPublic property or method — parent can set itParent → Child
@trackDeep reactivity for objects/arraysInternal
@wireBind a wire adapter (Salesforce data)Salesforce → Component

Lifecycle Hooks

These are the moments you'll care about most when testing:

HookWhen it firesCommon use
constructor()Component instantiatedRarely needed
connectedCallback()Added to DOMFetch data, set up listeners
renderedCallback()After each renderDOM manipulation (careful!)
disconnectedCallback()Removed from DOMCleanup listeners
Tester's tip

If a component behaves inconsistently on load, suspect connectedCallback(). Race conditions between async Apex calls and rendering are a common bug source in the portal.

Scoped CSS

CSS in an LWC only applies to that component. You can't accidentally override styles from another component. The platform adds a unique attribute to all elements at build time to enforce this.

/* Only affects elements inside THIS component */
.receipt-card {
  border: 1px solid #e0e0e0;
  border-radius: 8px;
  padding: 1rem;
}

/* Use :host to style the component root element */
:host {
  display: block;
}

Module 2

Data & Wire Adapters

Components need Salesforce data. There are two ways to get it: declaratively via @wire, or imperatively by calling an Apex method like a function. Each has trade-offs.

The @wire decorator

@wire subscribes a component to a data source. When the data changes, the component re-renders automatically. Think of it like a reactive database query.

import { LightningElement, wire, api } from 'lwc';
import { getRecord } from 'lightning/uiRecordApi';
import FIRST_NAME from '@salesforce/schema/Contact.FirstName';
import LAST_NAME from '@salesforce/schema/Contact.LastName';

export default class DonorHeader extends LightningElement {
  @api recordId;

  // Wire: auto-fetches Contact record when recordId is set
  @wire(getRecord, { recordId: '$recordId', fields: [FIRST_NAME, LAST_NAME] })
  contact;

  get fullName() {
    return `${this.contact?.data?.fields?.FirstName?.value} ${this.contact?.data?.fields?.LastName?.value}`;
  }
}
donorHeader.js
The $ prefix

'$recordId' means "reactively re-run this wire whenever recordId changes." Without the $, it runs once with the initial value.

Wire result shape

Wire adapters always return an object with data and error. Always check both.

// Two ways to receive wired data:

// 1. As a property (access via .data / .error)
@wire(getRecord, { recordId: '$recordId', fields: [FIRST_NAME] })
contact;  // this.contact.data, this.contact.error

// 2. As a function (called with result)
@wire(getRecord, { recordId: '$recordId', fields: [FIRST_NAME] })
wiredContact({ data, error }) {
  if (data) { this.donorName = data.fields.FirstName.value; }
  if (error) { console.error(error); }
}
donorHeader.js

Calling Apex directly (Imperative)

For operations like form submissions, button clicks, or anything that needs manual control — use imperative Apex.

import { LightningElement } from 'lwc';
import getGiftHistory from '@salesforce/apex/DonorPortalController.getGiftHistory';

export default class GiftHistory extends LightningElement {
  gifts = [];
  isLoading = false;
  error;

  connectedCallback() {
    this.loadGifts();
  }

  async loadGifts() {
    this.isLoading = true;
    try {
      this.gifts = await getGiftHistory();
    } catch (e) {
      this.error = e.body?.message;
    } finally {
      this.isLoading = false;
    }
  }
}
giftHistory.js

Wire vs Imperative — when to use each

@wireImperative
Best forRead-only data, auto-refreshActions, conditionally fetched data
TriggersAutomatically on param changeYou control when
Error handling.error propertytry/catch
In testsMock the adapterMock the import
⚠ Common bug in the portal

Wired calls run before connectedCallback(). If a component depends on wired data being ready inside connectedCallback(), it will be undefined. Always use the wired function form or a getter with null checks.

Module 3

Experience Cloud

Experience Cloud is the Salesforce platform for building external-facing sites — portals, communities, help centres. The Donor Portal is one of these sites. It changes how components behave in some important ways.

How the portal is structured

Experience Cloud Site (Donor Portal)
  └── Pages (Home, My Gifts, Tax Receipts, Profile)
      └── Regions (header, body, footer)
          └── LWC Components (dropped in via Builder)
              └── Child LWCs (composed inside components)

Data layer:  Salesforce Core Org <─── Experience Cloud fetches via Apex / Wire

Guest vs Authenticated Users

Experience Cloud has two types of page visitors. This matters enormously for testing.

TypeWhoRecord accessApex context
GuestNot logged inVery limited (public data only)Runs as Guest User profile
AuthenticatedLogged in donorTheir own recordsRuns as their Contact's linked User
⚠ Sharing rules matter

If a donor can't see their gifts, it's almost always a sharing or profile permissions issue — not a bug in the LWC code. Always check object/field permissions and sharing rules before debugging the component.

OTP Authentication (Donor Portal specific)

WFP's Donor Portal uses a passwordless One-Time Password (OTP) login flow. This is a custom LWC authentication sequence:

  1. Donor enters their email
  2. An Apex controller generates and sends an OTP via email
  3. Donor enters the code — a second Apex call validates it
  4. On success, the session is established and donor is redirected
Testing tip

OTP flows are time-sensitive. When testing, check if there's a backdoor test user or a bypass flag in the Custom Metadata. Never test with real donor emails in sandbox.

CurrentPageReference — knowing where you are

Components in Experience Cloud often need to know the current page or URL parameters. They use CurrentPageReference for this:

import { wire, LightningElement } from 'lwc';
import { CurrentPageReference } from 'lightning/navigation';

export default class DonorPortalRouter extends LightningElement {
  @wire(CurrentPageReference)
  pageRef;

  get currentSection() {
    return this.pageRef?.state?.section;
  }
}

NavigationMixin — navigating between pages

Components navigate programmatically using NavigationMixin:

import { LightningElement } from 'lwc';
import { NavigationMixin } from 'lightning/navigation';

export default class ReceiptButton extends NavigationMixin(LightningElement) {
  goToReceipts() {
    this[NavigationMixin.Navigate]({
      type: 'comm__namedPage',
      attributes: { name: 'Tax_Receipts__c' }
    });
  }
}

Module 4

Testing LWCs

Your testing background is your superpower here. LWC testing has two layers: automated unit tests with Jest, and manual/integration testing in the sandbox. Both matter.

Jest — Unit Testing LWCs

Salesforce provides @salesforce/sfdx-lwc-jest — a Jest preset that stubs Salesforce-specific imports so you can test components in Node.js without a Salesforce org.

// donorSummary.test.js
import { createElement } from 'lwc';
import DonorSummary from 'c/donorSummary';

describe('c-donor-summary', () => {

  // Always clean up after each test
  afterEach(() => {
    while (document.body.firstChild) {
      document.body.removeChild(document.body.firstChild);
    }
  });

  it('renders donor name when donorId is set', () => {
    // 1. Create component
    const el = createElement('c-donor-summary', { is: DonorSummary });
    el.donorId = '003xx000004TmiQAAS';
    document.body.appendChild(el);

    // 2. Wait for async rendering
    return Promise.resolve().then(() => {
      // 3. Assert DOM state
      const name = el.shadowRoot.querySelector('.donor-name');
      expect(name).not.toBeNull();
    });
  });
});
donorSummary.test.js

Mocking @wire adapters

Since Jest runs without a real Salesforce org, you mock wire adapters to simulate data and error states:

import { createElement } from 'lwc';
import { registerLdsTestWireAdapter } from '@salesforce/sfdx-lwc-jest';
import { getRecord } from 'lightning/uiRecordApi';
import DonorHeader from 'c/donorHeader';

const getRecordAdapter = registerLdsTestWireAdapter(getRecord);

const MOCK_CONTACT = {
  fields: { FirstName: { value: 'Amara' }, LastName: { value: 'Diallo' } }
};

it('displays donor full name from wire data', async () => {
  const el = createElement('c-donor-header', { is: DonorHeader });
  document.body.appendChild(el);

  // Push mock data into the wire adapter
  getRecordAdapter.emit(MOCK_CONTACT);

  await Promise.resolve();

  const heading = el.shadowRoot.querySelector('h1');
  expect(heading.textContent).toBe('Amara Diallo');
});

import getGiftHistory from '@salesforce/apex/DonorPortalController.getGiftHistory';

Manual Testing Checklist

For the Donor Portal, always test across these dimensions:

AreaWhat to check
Auth statesGuest (not logged in), just-logged-in, session expired mid-flow
Data statesFirst-time donor (no gifts), donor with 1 gift, donor with 50+ gifts
Error statesNetwork failure, Apex exception, validation error, timeout
LocalisationEN, FR, NL — check labels, date/currency formatting, RTL if applicable
Tax receiptsCA (T4a), NL (ANBI), FR — each has different generation logic
DevicesDesktop, tablet, mobile (portal is responsive)
Pro tip — Event testing

When a component fires a custom event, your test should assert that the event was dispatched with the right detail. Use addEventListener on the element before triggering the action.

it('fires receiptDownloaded event with correct id', async () => {
  const handler = jest.fn();
  el.addEventListener('receiptdownloaded', handler);

  el.shadowRoot.querySelector('button').click();

  await Promise.resolve();
  expect(handler).toHaveBeenCalledTimes(1);
  expect(handler.mock.calls[0][0].detail.receiptId).toBe('R-001');
});

Donor Portal Component Tour

Key Component Areas

Authentication Flow

  ├── donorEmailInput    ← validates email format

Gift History

  ├── giftHistoryTable   ← Apex: getGiftHistory(filters)

  • Test focus: Filter combinations, pagination, sort order, empty results

Tax Receipts

taxReceiptPage

  │   └── taxReceiptItem  ← download button → Apex: generateReceiptPDF()

  • Test focus: Different receipt templates by country (CA/NL/FR), download failure handling, year with no receipts

Debugging in the Portal

SymptomFirst place to look
Component doesn't renderBrowser console — LWC syntax errors surface here
Data doesn't loadNetwork tab → Salesforce API calls, then check Apex logs
Works for admin, not donorProfile/permission set on Experience Cloud site
Wrong language shownCustom Label translations in Setup, or user's locale setting
PDF doesn't generateApex logs → usually a template variable is null or unresolved
Apex Debug Logs

Go to Setup → Debug Logs, add a trace flag for the donor user, reproduce the issue, then check the log. Look for USER_DEBUG entries and any exceptions. The log level is usually set in the Custom Metadata.

LWC Cheat Sheet

Your quick-reference for the most common patterns you'll encounter in the Donor Portal codebase.

Decorators

@api donorId;           // public prop, parent sets this
@track gifts = [];      // deep reactive (objects/arrays)
@wire(getRecord, {...}) // reactive Salesforce data

Template directives

<!-- Conditional -->
<template if:true={showBanner}>...</template>
<template if:false={isLoading}>...</template>

<!-- Loop (key required!) -->
<template for:each={gifts} for:item="gift">
  <div key={gift.Id}>{gift.Name}</div>
</template>

<!-- Iteration (index available) -->
<template iterator:it={gifts}>
  <div key={it.value.Id}>#{it.index}: {it.value.Name}</div>
</template>

Custom Events (child → parent)

// Child fires event
this.dispatchEvent(new CustomEvent('receiptdownloaded', {
  detail: { receiptId: 'R-001' },
  bubbles: true,      // crosses shadow DOM
  composed: true
}));

// Parent listens
<c-receipt-button onreceiptdownloaded={handleReceiptDone}></c-receipt-button>

Lifecycle quick map

constructor()         → avoid, use connectedCallback
connectedCallback()   → fetch data, setup (runs once)
renderedCallback()    → after EVERY render, use sparingly
disconnectedCallback()→ cleanup timers/listeners

Common imports

import { LightningElement, wire, api, track } from 'lwc';
import { getRecord, updateRecord } from 'lightning/uiRecordApi';
