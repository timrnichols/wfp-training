LWC for Testers

This course gives you a working mental model of Lightning Web Components — specifically in the context of WFP's Donor Portal. You don't need Salesforce experience. You do need your existing testing instincts, which transfer well.

What you'll learn

The right mental model

If you've tested web applications before, you already understand components. Think of LWC like a React or Angular component — a self-contained unit with:

  • Template (HTML) — what renders in the browser
  • Controller (JS) — state and behaviour
  • Style (CSS) — scoped styles, never leak out
🌍 WFP Donor Portal — What it actually is

The portal is a set of externally-facing LWC pages backed by Nonprofit Cloud (NPC) on the Bu3 sandbox. It is not built on Experience Cloud — there's no Community, no Experience Builder, no Guest User profile. Instead, pages are served directly and donors authenticate via email OTP (no password — a 6-digit code, valid 10 min, 3 attempts max). Once authenticated, they can view active recurring gifts, increase their donation, update contact details, manage communication preferences, and download tax receipts (CA, NL, and FR formats). Every screen is an LWC, or a composition of several.

Module 1

LWC Fundamentals

Every LWC lives in a folder. That folder always contains three files with the same name. Understanding this structure is 80% of reading any component.

The Component Bundle

donorSummary/
  ├── donorSummary.html   ← template (what the user sees)
  ├── donorSummary.js    ← controller (logic & state)
  ├── donorSummary.css   ← scoped styles
  └── donorSummary.js-meta.xml ← metadata (where it can deploy)

The HTML Template

LWC templates always wrap in <template>. Inside, it's mostly regular HTML — but with a few Salesforce directives for conditionals and loops.

<template>
  <!-- Conditional rendering -->
  <template if:true={isLoggedIn}>
    <p>Welcome back, {donorName}</p>
  </template>

  <!-- Loop -->
  <template for:each={gifts} for:item="gift">
    <p key={gift.Id}>{gift.Amount__c}</p>
  </template>

  <!-- Event handler -->
  <button onclick={handleDownload}>Download Receipt</button>
</template>
donorSummary.html
Key pattern

Curly braces {variableName} bind to JS properties. If the JS property changes, the HTML updates automatically — no DOM manipulation needed.

The JavaScript Controller

Every JS file exports a class that extends LightningElement. Properties you want bound to the template just need to be declared — or decorated.

import { LightningElement, track, api, wire } from 'lwc';

export default class DonorSummary extends LightningElement {

  // @api — public prop, set by parent component
  @api donorId;

  // @track — reactive private state (object/array)
  @track gifts = [];

  // plain property — also reactive for primitives
  donorName = '';
  isLoggedIn = false;

  // Lifecycle hook — runs after component renders
  connectedCallback() {
    console.log('Component mounted');
  }

  // Event handler
  handleDownload() {
    // fire an event, call Apex, etc.
  }
}
donorSummary.js

Decorators at a glance

DecoratorPurposeDirection
@apiPublic property or method — parent can set itParent → Child
@trackDeep reactivity for objects/arraysInternal
@wireBind a wire adapter (Salesforce data)Salesforce → Component

Lifecycle Hooks

These are the moments you'll care about most when testing:

HookWhen it firesCommon use
constructor()Component instantiatedRarely needed

renderedCallback()After each renderDOM manipulation (careful!)
disconnectedCallback()Removed from DOMCleanup listeners
Tester's tip

If a component behaves inconsistently on load, suspect connectedCallback(). Race conditions between async Apex calls and rendering are a common bug source in the portal.

Scoped CSS

CSS in an LWC only applies to that component. You can't accidentally override styles from another component. The platform adds a unique attribute to all elements at build time to enforce this.

/* Only affects elements inside THIS component */
.receipt-card {
  border: 1px solid #e0e0e0;
  border-radius: 8px;
  padding: 1rem;
}

/* Use :host to style the component root element */
:host {
  display: block;
}

Module 2

Data & Wire Adapters

Components need Salesforce data. There are two ways to get it: declaratively via @wire, or imperatively by calling an Apex method like a function. Each has trade-offs.

The @wire decorator

@wire subscribes a component to a data source. When the data changes, the component re-renders automatically. Think of it like a reactive database query.

import { LightningElement, wire, api } from 'lwc';
import { getRecord } from 'lightning/uiRecordApi';
import FIRST_NAME from '@salesforce/schema/Contact.FirstName';
import LAST_NAME from '@salesforce/schema/Contact.LastName';

export default class DonorHeader extends LightningElement {
  @api recordId;

  // Wire: auto-fetches Contact record when recordId is set
  @wire(getRecord, { recordId: '$recordId', fields: [FIRST_NAME, LAST_NAME] })
  contact;

  get fullName() {
    return `${this.contact?.data?.fields?.FirstName?.value} ${this.contact?.data?.fields?.LastName?.value}`;
  }
}
donorHeader.js
The $ prefix

'$recordId' means "reactively re-run this wire whenever recordId changes." Without the $, it runs once with the initial value.

Wire result shape

Wire adapters always return an object with data and error. Always check both.

// Two ways to receive wired data:

// 1. As a property (access via .data / .error)
@wire(getRecord, { recordId: '$recordId', fields: [FIRST_NAME] })
contact;  // this.contact.data, this.contact.error

// 2. As a function (called with result)
@wire(getRecord, { recordId: '$recordId', fields: [FIRST_NAME] })
wiredContact({ data, error }) {
  if (data) { this.donorName = data.fields.FirstName.value; }
  if (error) { console.error(error); }
}
donorHeader.js

Calling Apex directly (Imperative)

For operations like form submissions, button clicks, or anything that needs manual control — use imperative Apex.

import { LightningElement } from 'lwc';
import getGiftHistory from '@salesforce/apex/DonorPortalController.getGiftHistory';

export default class GiftHistory extends LightningElement {
  gifts = [];
  isLoading = false;
  error;

  connectedCallback() {
    this.loadGifts();
  }

  async loadGifts() {
    this.isLoading = true;
    try {
      this.gifts = await getGiftHistory();
    } catch (e) {
      this.error = e.body?.message;
    } finally {
      this.isLoading = false;
    }
  }
}
giftHistory.js

Wire vs Imperative — when to use each

@wireImperative
Best forRead-only data, auto-refreshActions, conditionally fetched data
TriggersAutomatically on param changeYou control when
Error handling.error propertytry/catch
In testsMock the adapterMock the import
⚠ Common bug in the portal

Wired calls run before connectedCallback(). If a component depends on wired data being ready inside connectedCallback(), it will be undefined. Always use the wired function form or a getter with null checks.

Module 3

Portal Architecture & OTP Auth

The Donor Portal is not built on Salesforce Experience Cloud. It's a set of externally-facing LWC pages served from Salesforce, backed by Nonprofit Cloud (NPC). Authentication is a fully custom email OTP flow — no Salesforce login, no Community, no Guest User profiles.

How the portal is structured

External LWC Pages (Donor Portal)
  └── Pages (Landing/Login, Home, Increase Impact, Contact Details, Preferences, Receipts)
      └── LWC Components (composed and served directly)
          └── Child LWCs (nested inside parent components)

Data layer:  NPC Org (Bu3 Sandbox) <─── Apex controllers, wire adapters
Auth layer:  Custom OTP — no Experience Cloud, no Community User

OTP Authentication Flow

This is the most important flow to understand — every test session starts here. There is no password. The donor's identity is proven by their email inbox.

1. Donor enters email on landing page
   └── Apex: DonorAuthController.requestOtp(email)
       ├── Looks up Contact by email
       ├── Generates 6-digit OTP, stores hashed in org
       └── Sends branded OTP email (within 30 sec)

2. Donor enters OTP
   └── Apex: DonorAuthController.verifyOtp(email, code)
       ├── Validates code hash + expiry
       ├── Increments attempt counter (max 3)
       └── On success: creates session, redirects to /home

3. Session active
   └── Hashed_Contact_Id__c passed in URL — plaintext ID never exposed
   └── Session expires after 30 minutes of inactivity

OTP Rules — know these cold

SettingValueWhere configured
OTP code length6 digitsApex constant
OTP expiry10 minutesCustom Metadata: OTP_Expiry_Minutes__mdt
Max failed attempts3 — then lockoutCustom Metadata
Session timeout30 minutesCustom Metadata
Email delivery SLAWithin 30 secondsEmail deliverability setting
⚠ Testing the OTP flow

Never test with real donor emails. In Bu3 sandbox, use the 4 designated test users (Donor A–D) whose inboxes you have access to. Check with Tim whether there's a bypass flag in Custom Metadata for automated test runs — hardcoded OTP codes are sometimes enabled in sandbox.

URL Security — Hashed Contact IDs

After authentication, the portal passes donor context via a hashed Contact ID in the URL — never the raw Salesforce ID. This is tested in TC 02.09 and TC 10.01.

Security test tip (TC 10.01)

Copy the cid value and try to base64-decode or reverse it. It should be opaque. The Apex controller uses an org-specific salt in the hash — it cannot be decoded externally. If you can derive the real Contact ID from the URL, that's a Critical severity defect.

Session Expiry & Logout

Two scenarios to always test per TC 02.08 and 02.10:

  • Idle timeout — wait 30+ minutes, try to navigate. Should redirect to login with "session expired" message.
  • Logout — click logout, then hit browser back. Portal content must not be accessible — the back button should not restore the session.

What "Guest vs Authenticated" means here

StateWhat they seeApex context
Pre-OTPLanding page with email input onlyNo session — Apex has no donor context
AuthenticatedFull portal — their own records onlySession holds resolved Contact ID; all Apex runs in that donor's context
⚠ Sharing rules still matter

Even without Experience Cloud, if a donor can see their name but not their gift history, it's almost always sharing or FLS — not a bug in the LWC. The custom OTP session resolves to a Contact, and all Apex runs with with sharing enforced (TC 10.04). Check object/field permissions before debugging the component.

Custom Labels & Multilingual support

The portal supports English, French, and Dutch. All user-facing strings are stored in Translation__mdt Custom Metadata (not standard Salesforce Custom Labels in this implementation). The LWC reads them via a wire call to a translation Apex controller, passing the donor's Preferred_Language__c.

// Translation pattern in the portal
@wire(getTranslations, { language: '$language' })
wiredLabels({ data }) {
  if (data) this.labels = data;
}

// In template:
<p>{labels.home_welcome}</p>

Module 4

Testing LWCs

Your testing background is your superpower here. LWC testing has two layers: automated unit tests with Jest, and manual/integration testing in the sandbox. Both matter.

Jest — Unit Testing LWCs

Salesforce provides @salesforce/sfdx-lwc-jest — a Jest preset that stubs Salesforce-specific imports so you can test components in Node.js without a Salesforce org.

// donorSummary.test.js
import { createElement } from 'lwc';
import DonorSummary from 'c/donorSummary';

describe('c-donor-summary', () => {

  // Always clean up after each test
  afterEach(() => {
    while (document.body.firstChild) {
      document.body.removeChild(document.body.firstChild);
    }
  });

  it('renders donor name when donorId is set', () => {
    // 1. Create component
    const el = createElement('c-donor-summary', { is: DonorSummary });
    el.donorId = '003xx000004TmiQAAS';
    document.body.appendChild(el);

    // 2. Wait for async rendering
    return Promise.resolve().then(() => {
      // 3. Assert DOM state
      const name = el.shadowRoot.querySelector('.donor-name');
      expect(name).not.toBeNull();
    });
  });
});
donorSummary.test.js

Mocking @wire adapters

Since Jest runs without a real Salesforce org, you mock wire adapters to simulate data and error states:

import { createElement } from 'lwc';
import { registerLdsTestWireAdapter } from '@salesforce/sfdx-lwc-jest';
import { getRecord } from 'lightning/uiRecordApi';
import DonorHeader from 'c/donorHeader';

const getRecordAdapter = registerLdsTestWireAdapter(getRecord);

const MOCK_CONTACT = {
  fields: { FirstName: { value: 'Amara' }, LastName: { value: 'Diallo' } }
};

it('displays donor full name from wire data', async () => {
  const el = createElement('c-donor-header', { is: DonorHeader });
  document.body.appendChild(el);

  // Push mock data into the wire adapter
  getRecordAdapter.emit(MOCK_CONTACT);

  await Promise.resolve();

  const heading = el.shadowRoot.querySelector('h1');
  expect(heading.textContent).toBe('Amara Diallo');
});

For the Donor Portal, always test across these dimensions:

Pro tip — Event testing

When a component fires a custom event, your test should assert that the event was dispatched with the right detail. Use addEventListener on the element before triggering the action.

it('fires receiptDownloaded event with correct id', async () => {
  const handler = jest.fn();
  el.addEventListener('receiptdownloaded', handler);

  el.shadowRoot.querySelector('button').click();

  await Promise.resolve();
  expect(handler).toHaveBeenCalledTimes(1);
  expect(handler.mock.calls[0][0].detail.receiptId).toBe('R-001');
});

Lightning Studio — Your In-Browser Dev Tool

Install the Lightning Studio Chrome extension before you start testing. It runs directly against the Bu3 sandbox org and gives you access to component source, Apex, and live query tools without needing VS Code or a full dev setup.

Setup — 30 seconds

Install from the Chrome Web Store, then log in to Bu3 sandbox in the same browser. The Lightning Studio icon in the toolbar will turn blue when it's connected and ready. If it stays grey after install, refresh the Salesforce tab.

What you'll use it for

FeatureWhen to use it
Browse LWC sourceRead the HTML/JS/CSS of any portal component directly in the browser — no repo access needed. Useful for understanding what a component is supposed to render before filing a bug.
Browse Apex ClassesRead DonorAuthController, DonorPortalController, etc. to understand what the Apex is querying and what it returns — helps diagnose data issues.
Anonymous Apex (beta)Run a quick Apex snippet against the sandbox — e.g. query a specific Contact's Gift Commitments to verify test data before running TC 03.02.
REST API Explorer (beta)Call Salesforce REST endpoints directly to inspect raw response payloads — useful when a component shows unexpected data and you want to rule out the API layer.
SOQL to GraphQLWrite a SOQL query, run it, inspect results — faster than opening Salesforce Inspector or Developer Console for data checks.
Code Comparison (diff)Compare two versions of a component or Apex class — handy after Tim deploys a fix and you want to confirm what changed before re-running a failed test case.
Backup as ZIPDownload a component or full metadata backup — useful if you need a snapshot before a big deployment.
Tester's workflow with Lightning Studio

When a test case fails — say TC 04.04 (gift upgrade not persisting) — open Lightning Studio, find the Apex class handling the update, read the method, then use Anonymous Apex to query the Gift Commitment record directly. This tells you in under a minute whether the bug is in the LWC, the Apex, or the data — before writing a defect report.

⚠ Bu3 Sandbox only

Only use Lightning Studio against the Bu3 sandbox (wfp--bu3.sandbox.my.salesforce.com). Never connect it to a production org. The extension works purely locally — all data stays in your browser, nothing is sent to external servers.

LWC Cheat Sheet

Your quick-reference for the most common patterns you'll encounter in the Donor Portal codebase.

Decorators

@api donorId;           // public prop, parent sets this
@track gifts = [];      // deep reactive (objects/arrays)
@wire(getRecord, {...}) // reactive Salesforce data

Template directives

<!-- Conditional -->
<template if:true={showBanner}>...</template>
<template if:false={isLoading}>...</template>

<!-- Loop (key required!) -->
<template for:each={gifts} for:item="gift">
  <div key={gift.Id}>{gift.Name}</div>
</template>

<!-- Iteration (index available) -->
<template iterator:it={gifts}>
  <div key={it.value.Id}>#{it.index}: {it.value.Name}</div>
</template>

Custom Events (child → parent)

// Child fires event
this.dispatchEvent(new CustomEvent('receiptdownloaded', {
  detail: { receiptId: 'R-001' },
  bubbles: true,      // crosses shadow DOM
  composed: true
}));

// Parent listens
<c-receipt-button onreceiptdownloaded={handleReceiptDone}></c-receipt-button>

Lifecycle quick map

constructor()         → avoid, use connectedCallback
connectedCallback()   → fetch data, setup (runs once)
renderedCallback()    → after EVERY render, use sparingly
disconnectedCallback()→ cleanup timers/listeners

Common imports

import { LightningElement, wire, api, track } from 'lwc';
import { getRecord, updateRecord } from 'lightning/uiRecordApi';
import { ShowToastEvent } from 'lightning/platformShowToastEvent';
import LABEL from '@salesforce/label/c.Label_Name';
import myMethod from '@salesforce/apex/ClassName.methodName';
// Note: NavigationMixin / CurrentPageReference are Experience Cloud APIs
// and are NOT used in the Donor Portal (standalone LWC, no Community)

Toast notification

  title: 'Receipt downloaded',
  message: 'Your tax receipt is ready.',
